## 네트워크 계층 구조 파악

### 네트워크의 개념
> 네트워크는 원하는 정보를 원하는 수신자 또는 기기에 전송하기 위한 `인프라`이다.

### OSI(Opne system Interconnection) 7계층
- OSI 7계층 국제 표준화 기구인 ISO에서 개발한 `컴퓨터 네트워크 프로토콜 디자인`과 `통신`을 계층으로 나누어
설명한 `개방형 시스템 상호 연결 모델`
- 각 계층은 하위계층의 `기능을 이용`하여 상위 계층에 기능을 `제공한다`
    - -> 하위 ---> 상위 | 이용
- 1계층인 물리 계층부터 7계층인 애플리케이션 계층로 정의 되어있다.
- 각 계층을 지날 때마다 `헤더(Header)`가 붙는데, 계층의 기능과 관련된 제어 정보가 포함되어있다.
- 각 제어 정보들은 모두 운영체제가 제공하는 프로토콜에 의해 
- 송신 측에서는 `계층을 지날 때마다` 추가되고, 수신 측에서는 `제거` 된다.

### OSI 7 Layer의 특징

| 계층 이름 | 설명 | 프로토콜 | 전송단위 | 장비 |
|---|---|---|---|---|
| 응용 계층 (Application Layer) | 사용자와 네트워크 간 응용서비스 연결, 데이터 생성 | HTTP, FTP | 데이터 (Data) | 호스트(PC 등) |
| 표현 계층 (Presentation Layer) | 데이터 형식 설정, 부호교환, 암·복호화, 압축 | JPEG, MPEG | 데이터 (Data) | 호스트(PC 등) |
| 세션 계층 (Session Layer) | 송수신 간의 논리적인 연결, 연결 접속·동기제어 | RPC, NetBIOS | 데이터 (Data) | 호스트(PC 등) |
| 전송 계층 (Transport Layer) | 송수신 프로세스 간의 연결, 신뢰성 있는 통신 보장,<br> 데이터 분할·재조립, 흐름 제어, 오류 제어, 혼잡 제어 | TCP, UDP | 세그먼트 (Segment) | L4 스위치 |
| 네트워크 계층 (Network Layer) | 단말기 간 데이터 전송을 위한 최적화된 경로 제공 | IP, ICMP | 패킷 (Packet) | 라우터 |
| 데이터링크 계층 (Data Link Layer) | 인접 시스템 간 데이터 전송, 전송 오류 제어, 동기화,<br> 오류 제어, 흐름 제어, 회선 제어 | HDLC, PPP | 프레임 (Frame) | 브리지, 스위치 |
| 물리 계층 (Physical Layer) | 0과 1의 비트 정보를 회선에 보내기 위한 전기적 신호로 변환 | RS-232C | 비트 (Bit) | 허브, 리피터 |

<details open>
    <summary>위 특징 중 모르는 용어 정리</summary>

> 프로토콜에서 처음보는 단어가 많았고 해당 용어가 실제 프로젝트에서 어디에 쓰이는지를 중점적으로 찾아봄

- FTP(File Transfer Protocol)
    - 파일전송용 프로토콜 ("엣날부터 쓰이는 파일 전송/업로드) 
    - 쓰이는 시점 : 서버 ↔ 서버, 서버 ↔ 외부기관 사이에 `“파일을 주고받는”` 통합에서 종종 등장
    - 프로젝트 적용
        - 레거시/기관 연동 : `매일 자정에 정산 파일 내려받기,배치파일 로그 업로드`
        - 다만 요즘은 보안 때문에 SFTP(SSH 기반) 또는 HTTPS 기반 업로드(API)로 대체되는 경우가 많음

- MPEG(Moving Picture Experts Group)
    - 영상(멀티미디어) 압축/인코딩 표준 계열(예: MPEG-2, MPEG-4)
    - 흔히 말하는 mp4도 MPEG-4 기반 컨테이너/코덱과 엮인다.
    - 언제 쓰냐: 영상 저장/스트리밍/업로드 처리.
    - 프로젝트에서 어떻게 쓰이냐
        - 사용자가 영상 업로드 → 서버에서 트랜스코딩(화질 변환) → HLS/DASH로 스트리밍
        - 백엔드 입장에선 직접 `“MPEG 프로토콜”을 구현`한다기보다,<br>
        `FFmpeg, MediaConvert(AWS) 같은 도구/서비스`로 “인코딩 규격”을 다루게 됩니다.
    - 키 포인트: “네트워크 프로토콜”이라기보다 미디어 포맷/압축 규격으로 이해하면 딱 맞습니다.

- UDP
    - 뭐냐: 전송계층 프로토콜. 연결 없음, 빠름, 대신 신뢰성 보장(재전송/순서보장) 없음.
    - 언제 쓰냐: “조금 손실돼도 되는 대신 지연이 치명적인” 경우
    - 실시간 스트리밍/게임/음성통화/영상통화/라이브
    - 프로젝트에서 어떻게 쓰이냐
        - 백엔드 API 서버에서는 보통 TCP(HTTP)가 대부분이라 UDP를 직접 다룰 일이 적지만,
        - 아래는 간접적으로 정말 자주 만납니다:
        - DNS가 UDP 기반(기본 쿼리)
        - QUIC/HTTP3는 UDP 위에서 동작
    - 키 포인트: “빠른 대신 믿지 말 것” → 필요하면 애플리케이션 레벨에서 보완.

- ICMP
    - 뭐냐: IP 네트워크의 상태/오류 메시지를 주고받는 프로토콜.
    - 언제 쓰냐: 네트워크가 “살아있나?”, “경로가 어디서 막히나?” 같은 진단.
    - 프로젝트에서 어떻게 쓰이냐
        - ping = ICMP Echo Request/Reply
        - traceroute/tracert도 ICMP(또는 UDP 변형)로 경로 추적
        - 운영에서 “서버 접속 안 됨” 이슈 나면 제일 먼저 ICMP로 생존/경로 확인합니다.
    - 키 포인트: 보안상 ICMP를 막는 환경도 많아서, ping 안 된다고 무조건 죽었다고 단정하면 안 됩니다.

- HDLC
    - 뭐냐: 데이터링크 계층의 점대점(PTP) 통신용 프레이밍(프레임 구조) 프로토콜.
    - 주로 WAN/전용선 같은 환경에서 쓰던 전통 기술.
    - 언제 쓰냐: 통신사 전용 회선, 라우터 간 링크 같은 “저수준 링크” 구성에서.
    - 프로젝트에서 어떻게 쓰이냐
        - 일반 웹/앱 개발자는 직접 다룰 일이 거의 없고,
        - 네트워크 장비(라우터/스위치) 설정이나 통신 인프라 쪽에서 등장합니다.
    - 키 포인트: “개발자가 코드로 쓰는 프로토콜”이라기보다 네트워크 장비 레벨의 링크 기술.
- PPP
    - 뭐냐: 데이터링크 계층에서 점대점 링크에 쓰는 프로토콜.
    - 예전에 전화모뎀/ADSL 시절, “PPP 접속” 같은 표현 많이 들어보셨을 거예요.
    - 언제 쓰냐: ISP 접속(특히 PPPoE), 인증/세션 형태의 점대점 연결.
    - 프로젝트에서 어떻게 쓰이냐
        - 이것도 대부분 통신/네트워크 인프라 쪽이고,
        - 개발자는 보통 “집 인터넷이 PPPoE다” 정도로 간접 체감합니다.
    - 키 포인트: HDLC와 비슷하게 링크 계층에서 네트워크를 “태워주는” 역할.
- RS-232C
    - 뭐냐: 물리 계층(케이블/전기 신호 수준)의 직렬 통신 규격. 흔히 “시리얼 포트” 통신.
    - 언제 쓰냐: 산업/임베디드/장비 제어(PLC, 센서, 계측기, POS 장비 등)
    - 프로젝트에서 어떻게 쓰이냐
        - 서버 프로젝트에서도 “장비 연동”이 들어가면 갑자기 등장합니다.
        - 예: 공장 설비 데이터 수집 → 게이트웨이(라즈베리파이/산업용 PC)가 RS-232로 읽음 → MQTT/HTTP로 서버로 올림
    - 키 포인트: 백엔드라 해도 “현장 장비 연동” 하면 만날 확률이 올라갑니다.
</details>
