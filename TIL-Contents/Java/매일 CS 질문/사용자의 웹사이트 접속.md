# 사용자가 웹사이트에 처음 접근했을 때 발생하는 일련의 과정에 대해 설명해 주세요.

> 내 답변 : 약간 AWS에서 배포 흐름 생각해서 답변해 봄

리엑트 기반 프론트와 + Springboot 백엔드 코드의 경우를 설명하겠습니다. 

먼저 웹사이트가 어떻게 올라가는지를 알아야합니다.
웹사이트는 보통 정적 화면을 구성하는 코드를 기반으로 도메인이 올라가게 되는데

사용자가 그 도메인에 접속 시 정적으로 구성된 화면을 보게됩니다
거기서 사용자가 어떠한 기능들을 시행하게 되면 
해당 컴포넌트가 정적기능인지 동적 기능인지 CloudFront의 판단에 의해 
백엔드 서버가 호출이 되고 API기능을 수행하여 필요 시 DB기능까지 수행 후 
응답을 하게됩니다.

따라서 사용자가 웹사이트에 처음 접근하게 되면 도메인에 대한 사용자 이용이 발생한것이기 때문에 비용이 발생합니다.

---

> 정석 답변 : 

- 사용자가 웹사이트 도메인에 접속하면 먼저 DNS를 통해 도메인에 대한 IP(또는 CDN 엔드포인트)를 조회합니다.
- 이후 브라우저는 해당 서버와 TCP 연결을 맺고, HTTPS라면 TLS 핸드셰이크로 암호화 통신을 준비합니다.
- 연결이 준비되면 브라우저는 GET / 요청을 보내고, `보통 CDN/정적 호스팅에서 index.html과 JS/CSS 같은 정적 리소스`를 내려받습니다.
- React SPA의 경우 내려받은 JS 번들이 실행되며 화면을 렌더링하고, 
화면에 필요한 데이터가 있으면 fetch/axios로 /api 요청을 추가로 보내 Spring Boot 백엔드가 처리합니다.
- 백엔드는 로직 수행 중 필요하면 DB를 조회/갱신하고 JSON 응답을 반환합니다.
- 이 과정에서 요청 수, 데이터 전송량, 서버/DB 사용량에 따라 CDN·로드밸런서·서버·DB 비용이 발생할 수 있습니다.

<details>
    <summary>단계별 설명(이게 더 좋음)</summary>

1) DNS 조회
- 사용자가 브라우저에 www.example.com 입력
- 브라우저/OS가 DNS 캐시 확인 → 없으면 **리졸버(통신사/공용 DNS)**로 질의
- 최종적으로 도메인에 대한 **IP 주소(또는 CloudFront 같은 CDN 주소)**를 얻음
2) TCP 연결 + (HTTPS면) TLS 핸드셰이크
- 브라우저가 해당 IP로 TCP 3-way handshake로 연결 수립
- HTTPS면 이어서 TLS 핸드셰이크로 암호화 통신 준비(인증서 검증 포함)
3) HTTP 요청/응답: “최초 문서” 받기
- 브라우저가 GET / 요청
- 보통 CDN/정적 호스팅(S3/CloudFront 등)에서 index.html 응답
4) 정적 리소스 로딩
- 브라우저가 HTML을 파싱하면서 필요한 CSS/JS/이미지를 추가로 요청
- JS(React 번들)가 로드되고 실행됨
5) React 앱 구동(렌더링) + API 호출
- React가 라우팅/렌더링을 수행
- 화면에 필요한 데이터가 있으면 fetch/axios로 백엔드 API 호출 (/api/...)
- Spring Boot가 요청을 처리하며 필요 시 DB 조회/저장 후 JSON 응답
6) 캐싱/최적화 포인트(가산점)
- CDN 캐시, 브라우저 캐시, Keep-Alive, HTTP/2/3, gzip/brotli, ETag/Last-Modified 같은 걸 한 줄이라도 붙이면 “CS 이해도”가 확 올라갑니다.
</details>
